# 云端账本 — 产品需求文档（PRD）

> **项目名称**：云端账本（Fluid Ledger）  
> **文档版本**：v1.2  
> **文档日期**：2026-04-14  
> **目标平台**：iOS 移动端、Android 移动端、微信小程序  
> **设计基准分辨率**：428 × 926 px（iPhone 12 Pro Max）  
> **文档状态**：需求细化稿（已补充范围边界与业务规则）

## 一、项目概述

### 1.1 产品定位

云端账本是一款面向 **45 岁以上线下个体小生意人**（农资店、小卖部、小吃店、批发零售摊贩等）的 AI 辅助智能记账应用。核心解决以下痛点：

| 痛点 | 解决方案 |
|------|---------|
| 手工记账易错、效率低 | 语音优先的 AI 对话式录入，自动生成账目 |
| 复杂财务软件难上手 | 极简操作，核心流程不超 2 步，无专业财务术语 |
| 库存/欠款算不清 | 智能库存自动同步，待收/待付明细自动汇总 |
| 盈利状况不明 | 自动核算利润，大白话版收支报表 |
| 客户赊账记不住 | AI 自动关联客户，到期自动催款提醒 |
| 纸质账本怕丢失 | 云端加密自动备份，多端实时同步 |

### 1.2 目标用户画像

| 维度 | 描述 |
|------|------|
| **年龄段** | 45 岁以上 |
| **职业** | 个体工商户、小摊贩、小店主（农资店、小卖部、小吃店、批发零售摊贩等） |
| **技术水平** | 对智能手机操作不太熟练，不习惯打字，但熟悉微信语音 |
| **核心诉求** | 简单、快速、看得清、搞得懂 |
| **使用场景** | 进货时随手记录 → 卖货时语音入账 → 月底看报表 → 赊账到期催款 |
| **设备环境** | 中低端安卓手机为主，部分使用 iPhone，网络条件参差不齐 |

### 1.3 设计原则

1. **语音优先**：所有核心记账操作均可通过语音完成，符合目标用户"说"多于"打"的习惯
2. **大字体大按钮**：全界面采用高辨识度视觉元素，最小可交互区域 48×48px
3. **零门槛**：无专业财务术语，所有展示用大白话表达
4. **极简操作**：核心操作不超过 2 步，减少用户决策负担
5. **无干扰**：零广告弹窗、零冗余功能，聚焦核心价值
6. **数据安全**：云端加密备份，多种生物识别保护
7. **离线可用**：弱网/断网环境下核心记账功能可离线使用，联网后自动同步
8. **容错友好**：所有录入操作均可撤销、修改，AI 识别结果必须用户确认后才写入

### 1.4 当前版本范围定义

> 以下范围用于解决本文档中“全量信息架构”与“首发版本开发范围”之间的差异；首发排期、原型补齐和测试验收均以本节为准。

| 模块 | v1.0 范围定义 | 说明 |
|------|--------------|------|
| 账号与店铺初始化 | **必做** | 手机号验证码登录/注册、小程序微信授权登录、首次填写店铺基础信息 |
| 启动与安全验证 | **必做** | 启动页、生物识别/图案/密码验证、失败锁定、密码兜底 |
| 首页 AI 记账 | **必做** | 语音录入、文字录入、AI 确认卡片、基础多轮补问 |
| 拍照识别 | **v1.0 可降级** | 首发可先支持拍照上传 + 识别确认；相机实时增强能力可后续优化 |
| 库存管理 | **必做** | 库存总览、搜索筛选、商品详情、增减调整、低库存/临期提醒 |
| 账本总览 | **必做** | 待收/待付概览、利润概览、资金趋势、AI 建议 |
| 待收/待付明细 | **必做** | 单条催款、标记已收/已付、基础筛选；批量操作可后续增强 |
| 交易流水与交易详情 | **v1.0 可选** | 若首发周期受限，可暂以后续迭代补充，不阻塞核心原型页面上线 |
| 收支报表查看 | **v1.0 可选** | 首发可仅保留账本总览；日/月/年报表页面可后续补齐 |
| 报表导出（Excel/PDF） | **v2.0** | 不进入首发 |
| 客户/供应商独立管理 | **v2.0** | v1 仅在记账过程中自动创建或关联往来对象，不提供独立管理入口 |
| 新手引导 | **v2.0** | v1 首装可直接走登录/初始化流程，不强依赖三步引导 |

**首发版本判断原则**：
- 以原型中的核心页面链路为主：启动/登录 → 首页记账 → 库存 → 账本 → 设置/安全。
- 凡是不会阻塞“记一笔、查库存、看待收待付、备份数据”主链路的能力，可放入 v1.0 可选或 v2.0。
- 页面清单中的“全量信息架构”不等于“首发必做页面”，后文如有冲突，以本节和“十一、版本规划”为准。

### 3.3 P0 — 启动欢迎页

**页面定位**：App 启动后首先展示的品牌闪屏页面。

**功能要求**：
- 展示应用品牌 Logo 和名称"云端账本"
- 展示简短 Slogan（如"让记账像说话一样简单"）
- 自动加载初始化数据（本地缓存、登录态校验等）
- 加载完成后自动跳转：
  - 若已设置安全验证 → 跳转对应解锁页（P5-1/P5-2/P5-3）
  - 若已登录但未设置安全验证 → 直接进入首页（P1）
  - 若首次安装或无登录态 → 进入登录/注册流程（P-L1）
  - 若后续版本启用新手引导，则在首登成功后进入 P-G1~P-G3

**交互要求**：
- 停留时间不超过 3 秒
- 展示加载进度动画（可选）

| 底部交互栏 | 键盘切换按钮、语音录入按钮（主按钮）、拍照按钮 |

#### 3.4.2 语音录入功能

- **入口**：底部交互栏正中的醒目圆形麦克风大按钮（直径约 84px）
- **交互流程**：
  1. 用户点击/长按麦克风按钮开始录音
  2. 展示录音中的视觉反馈（脉冲动画、波形等）
  3. 用户松开或再次点击结束录音
  4. AI 实时转写语音为文字，以用户气泡形式展示在对话区
  5. AI 分析语音内容，自动识别意图（进货、销货、收款、赊账等）
  6. AI 以回复气泡 + 动态确认卡片形式展示识别结果

- **语音识别能力要求**：
  - 支持普通话，兼容常见方言口音
  - 能识别商品名称、数量、单位、金额等关键信息
  - 示例："买了3箱苹果，每箱88块钱" → 自动识别为入库操作

#### 3.4.3 AI 智能确认卡片

当 AI 识别到有效的记账信息后，在对话区生成结构化确认卡片：

- **卡片内容**：
  - 操作类型标签（如"库存确认"、"收入记录"、"支出记录"等）
  - 商品/项目名称（大字号突出显示）
  - 状态标签（如"待确认"）
  - 关键数据网格：数量、单价（两列布局）
  - 总金额（最大字号醒目展示）
  - 确认操作按钮（大尺寸，高度约 72px）

- **交互逻辑**：
  - 用户点击"确认入库"/"确认记账"后，数据正式写入
  - 确认后 AI 自动同步更新库存和账本数据
  - AI 会在确认后给出补充说明（如"已自动更新库存量并计入本月支出账单"）

#### 3.4.4 键盘输入

- 底部交互栏左侧提供键盘切换按钮
- 点击后弹出文字输入框，替代语音输入
- 输入内容同样通过 AI 分析生成账目

#### 3.4.5 拍照识别

- 底部交互栏右侧提供拍照按钮

## 五、AI 核心能力需求

### 5.1 语音识别与意图理解

| 能力 | 要求 |
|------|------|
| 语音转文字 | 支持普通话，兼容常见方言口音 |
| 意图识别 | 准确区分进货、销货、收款、付款、赊账等操作类型 |
| 实体提取 | 识别商品名称、数量、单位、单价、总价、客户名等 |
| 上下文理解 | 支持多轮对话修正和补充信息 |
| 容错处理 | 信息不完整时主动追问，不会错误记录 |

### 5.2 语音识别示例

| 用户语音 | AI 识别结果 |
|---------|------------|
| "买了3箱苹果，每箱88块钱" | 入库：苹果 × 3箱，单价 ¥88，总计 ¥264 |
| "卖了5袋大米，一袋60" | 出库/销售：大米 × 5袋，单价 ¥60，总计 ¥300 |
| "老张赊了200块钱的化肥" | 赊账：客户"老张"，商品"化肥"，金额 ¥200 |
| "收到李姐还的500块" | 收款：客户"李姐"，金额 ¥500 |

### 5.3 拍照识别

- 支持识别手写账单、打印小票、发票等
- 自动提取关键信息生成确认卡片
- 识别不清时支持手动修正

### 5.4 智能经营建议

- 基于经营数据生成通俗易懂的优化建议
- 建议类型：催款提醒、补货建议、利润分析、趋势预警等
- 所有用语使用大白话，避免专业财务术语

### 5.5 AI 置信度分级与兜底流程

| 置信度等级 | 系统行为 | 用户操作要求 |
|-----------|---------|-------------|
| 高 | 直接生成完整确认卡片 | 用户一键确认后写入 |
| 中 | 高亮不确定字段，并主动追问 1 个最关键问题 | 用户补充后再次确认 |
| 低 | 不允许直接写入，只展示“我没太听清”及候选结果 | 用户改用文字/手动表单/重新录音 |

**统一规则**：
- AI 识别结果**永不自动入账**，必须由用户明确确认
- 同名商品或同名客户命中多个候选时，必须先让用户选择
- 连续 3 次识别失败时，默认推荐切换为手动录入
- 对于数量、金额、往来对象、到期日等关键字段，缺失任一项时不得生成可提交的最终账目

### 5.6 AI 异常与权限处理

- 麦克风权限被拒绝：展示权限说明，并提供“改用文字输入”
- 相机权限被拒绝：展示权限说明，并提供“从相册上传”
- 网络异常：优先保存本地草稿，恢复联网后再继续识别
- 识别结果与库存/客户主数据冲突：优先提示用户选择，不做静默覆盖
- 重复提交风险：同一设备在短时间内提交内容高度相似的账目时，系统提示“这笔账可能已经记过了”

## 六、业务规则与口径定义

### 6.1 交易类型与联动规则

| 交易类型 | 典型口述 | 库存影响 | 账本影响 | 往来对象规则 |
|---------|---------|---------|---------|-------------|
| 进货现付 | 买了 3 箱苹果，付了 264 | 增加库存 | 记一笔支出 | 供应商可选 |
| 进货赊购 | 从老李那进了 10 袋化肥，先记着 | 增加库存 | 新增应付账款 | 供应商必填 |
| 销售现收 | 卖了 5 袋大米，收了 300 | 减少库存 | 记一笔收入 | 客户可选 |
| 销售赊销 | 老张赊了 200 块钱的化肥 | 减少库存 | 新增应收账款 | 客户必填 |
| 收款 | 收到李姐还的 500 块 | 不变 | 冲减应收，不重复记收入 | 客户必填 |
| 付款 | 给王老板结了 800 货款 | 不变 | 冲减应付，不重复记支出 | 供应商必填 |
| 盘点调整/报损 | 鸡蛋坏了 6 个 | 增减库存 | 记库存调整或损耗支出 | 无 |
| 退货/退款 | 退了 2 箱苹果给供货商 | 反向调整库存 | 冲减原交易金额 | 默认要求关联原交易 |

### 6.2 字段标准与校验规则

- 金额统一使用人民币，保留 2 位小数
- 数量统一使用正数录入，增减方向由交易类型决定
- 支持重量、件数、箱数等单位，数量最多保留 3 位小数
- 交易时间默认取用户确认时的本地时间，允许手动修改
- 数量、单价、总金额三者中若仅缺 1 项，系统可自动推导；若缺 2 项及以上，AI 必须追问
- 涉及应收/应付的交易，往来对象和到期日为必填
- 商品名称命中多个已有商品时，先提示“请选择要记的商品”
- 库存不足时默认提示风险；用户二次确认后允许继续记账，并将该记录标记为“库存异常”
- 已确认账目默认不做物理删除，首发阶段通过“作废/撤销”方式处理，并保留操作痕迹

### 6.3 利润、资金与报表口径

- **资金趋势**：按实际现金流统计，包含现收、收款、现付、付款、退款等现金变动
- **待收/待付**：按已确认但未结清的应收/应付款余额统计
- **利润概览**：按经营利润口径统计，不以现金是否到账为唯一标准
- 销售现收与销售赊销均计入收入；收款仅冲减应收，不重复计入收入
- 进货现付与进货赊购均形成库存成本；付款仅冲减应付，不重复计入支出
- 首发阶段销售成本按商品**移动平均成本**估算；若商品缺少成本数据，则在利润模块中标记为“成本待补全”
- 报表统计日界线采用店铺本地时区，自然日 00:00 至 23:59 计算

### 6.4 首发关键异常流程

- AI 无法识别商品：提示用户从已有商品中选择，或快速新建商品
- AI 无法识别往来对象：提示用户从历史客户/供应商中选择，或临时新增名称
- 同一笔账确认后发现录错：支持在结果 Toast 期间撤销，超时后通过“作废/修正”处理
- 多端同时修改同一笔账：不做静默合并，按同步策略生成冲突提示
- 催款发送失败：保留催款文案，不影响待收账款本身状态

## 七、数据与安全需求

### 7.1 数据存储

| 数据类型 | 存储方式 |
|---------|---------|
| 账目记录 | 云端加密存储 + 本地缓存 |
| 库存数据 | 云端加密存储 + 本地缓存 |
| 客户信息 | 云端加密存储 |
| 用户设置 | 云端同步 |
| 语音记录 | 临时存储，处理完成后删除原始音频 |

### 7.2 云端备份

- 自动备份：每日自动增量备份
- 手动备份：用户可随时触发全量备份
- 备份加密：使用 AES-256 加密
- 多端同步：同一账户多设备数据实时同步

### 7.3 离线与同步策略

- 首发采用 **Local First** 策略：用户确认后的账目先写入本地，再进入同步队列
- 每条记录至少包含以下同步状态：`待同步`、`同步中`、`同步成功`、`同步失败`
- 弱网或断网时，首页记账、库存调整、待收待付状态变更允许本地先完成
- 网络恢复后按“主数据优先、交易数据随后”的顺序重放同步队列
- 同一条记录在多端发生修改冲突时，不做静默字段合并；服务端保留最新确认版本，并生成冲突提示供用户人工确认
- 同步失败不丢数据，需支持用户手动重试
- 页面内需有轻量同步提示，避免用户误以为数据已完全上云

### 7.4 备份与恢复策略

- 自动增量备份默认保留近 30 天
- 手动全量备份默认保留最近 10 个版本
- 恢复前自动创建一次“恢复前快照”，用于回退
- 恢复范围包括：账目记录、库存数据、往来对象、店铺设置
- 不恢复设备级本地能力配置，如本地生物识别授权状态
- 恢复成功后需强制触发一次全量同步校验，确保当前设备与云端状态一致

### 7.5 安全认证

- 支持三种生物识别方式：面容识别（Face ID）、指纹识别、图案解锁
- 支持密码登录作为兜底方案
- 敏感操作（如删除账目、导出数据）需二次确认
- 连续 5 次验证失败锁定 30 分钟

### 7.6 隐私保护

- 所有数据传输使用 HTTPS/TLS 加密
- 不向第三方共享用户经营数据
- 本地生物特征数据不上传云端
- 遵守相关隐私法规（《个人信息保护法》等）
