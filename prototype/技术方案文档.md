## 二、总体技术结论

### 2.1 技术选型总览

| 领域 | 选型 | 说明 |
|------|------|------|
| 移动端主方案 | `uni-app x` | 一套代码编译到 iOS / Android / 微信小程序 |
| App 端兜底方案 | `SwiftUI` / `Jetpack Compose` | 若 POC 后某个 App 端体验不达标，可单端原生化 |
| 后端主服务 | `Java 21 + Spring Boot 3.5.x` | 模块化单体，承接核心业务逻辑 |
| AI 子服务 | `Python 3.12 + FastAPI` | 负责语音转写、OCR、LLM 解析编排 |
| 主数据库 | `PostgreSQL 17+` | 强事务、强关系、适合账本/库存场景 |
| 缓存 | `Redis 8` | 验证码、会话、幂等键、限流、热点缓存 |
| 消息队列 | `RabbitMQ 4` | 异步任务、通知、AI 回调、备份任务 |
| 对象存储 | `MinIO / S3 兼容存储` | 图片、语音、附件上传 |
| 数据迁移 | `Liquibase` | 库表结构版本管理 |
| API 描述 | `OpenAPI 3.1` | 接口文档、前后端联调契约 |
| 可观测性 | `OpenTelemetry + Prometheus + Grafana` | 日志、指标、链路追踪 |

### 2.2 关键技术决策

1. **不采用 uniCloud 的云函数、云对象作为正式项目主后端**
2. **客户端与后端完全解耦，统一走独立 API**
3. **首发采用模块化单体，不上微服务**
4. **AI 能力独立为子服务，不与核心账本服务强耦合**
5. **客户端支持离线缓存与同步队列，但业务真相源仍以后端数据库为准**
6. **先用 `uni-app x` 跑三端 POC，若某个 App 端体验不达标，再按端原生化**

## 三、总体技术架构

### 3.1 系统架构图

```text
+--------------------------------------------------------------+
|                        移动端/小程序层                        |
|  uni-app x                                                  |
|  - iOS App                                                  |
|  - Android App                                              |
|  - 微信小程序                                               |
+-------------------------------+------------------------------+
                                |
                                | HTTPS / JSON / JWT
                                v
+--------------------------------------------------------------+
|                        业务后端层                             |
|  Spring Boot API Server                                      |
|  - auth                                                      |
|  - shop                                                      |
|  - inventory                                                 |
|  - ledger                                                    |
|  - receivable/payable                                        |
|  - file                                                      |
|  - sync                                                      |
|  - backup                                                    |
|  - notification                                              |
|  - audit                                                     |
+-------------------+----------------+-------------------------+
                    |                |
                    | HTTP / MQ      | AMQP
                    v                v
         +------------------+   +------------------+
         |   AI Service     |   |   RabbitMQ       |
         | FastAPI Worker   |   | 异步任务队列      |
         +------------------+   +------------------+
                    |
                    v
         +------------------+
         | OCR / STT / LLM  |
         | 第三方 AI 能力    |
         +------------------+

+--------------------------------------------------------------+
|                         数据与基础设施层                      |
|  PostgreSQL | Redis | MinIO/S3 | Prometheus | Grafana        |
+--------------------------------------------------------------+
```

### 3.2 架构说明

- 移动端统一由 `uni-app x` 承担，减少三端首发的人力成本
- 核心账本、库存、待收待付必须在同一事务边界内完成，因此首发后端采用**模块化单体**
- AI 解析是“增强能力”，不是账本真相源，因此拆成独立 AI 子服务
- 图片、语音等大文件统一放对象存储，数据库只保存元数据
- 异步任务通过 `RabbitMQ` 解耦，例如：
  - AI 解析任务
  - 催款通知
  - 备份任务
  - 审计日志归档

## 四、移动端技术架构

### 4.1 客户端分层

```text
UI 页面层
  -> 领域交互层（记账、库存、待收待付）
    -> 平台能力适配层（录音、拍照、生物识别、上传）
      -> API Client / Sync Queue
        -> 本地缓存
```

### 4.2 客户端模块建议

- `pages/auth`：登录、验证码、店铺初始化
- `pages/home`：AI 记账首页、录音、拍照、确认卡片
- `pages/inventory`：库存列表、详情、编辑、筛选
- `pages/ledger`：账本总览、待收、待付
- `pages/settings`：店铺设置、安全设置、备份恢复
- `common/api`：接口请求封装
- `common/sync`：离线队列、同步状态管理
- `common/adapters`：录音、相机、权限、上传等平台差异层
- `common/models`：客户端 DTO 与领域模型

### 4.3 客户端边界原则

- 页面层不直接拼接后端 URL
- 页面层不直接处理签名、重试、幂等
- 平台差异必须收敛到 `adapter` 层
- 本地缓存只用于：
  - 离线可用
  - 列表加速
  - 待同步队列
- 本地缓存不作为最终账本真相源

### 4.4 移动端演进策略

- **首发策略**：三端统一由 `uni-app x` 输出
- **POC 验证期重点**：
  - iOS 输入框与滚动体验
  - 长列表流畅度
  - 录音、拍照、上传链路
  - 微信小程序样式与输入行为差异
- **兜底策略**：
  - 若 iOS 体验不达标：iOS 改为 `SwiftUI`
  - 若 Android 体验不达标：Android 改为 `Jetpack Compose`
  - 微信小程序保留 `uni-app x` 编译版本

## 五、后端技术架构

### 5.1 后端总体方案

后端采用 **模块化单体**，模块间通过清晰的包结构和服务边界管理复杂度，不在首发阶段引入微服务拆分。

### 5.2 后端核心模块

| 模块 | 职责 |
|------|------|
| `auth` | 手机号登录、验证码、刷新令牌、设备会话 |
| `shop` | 店铺资料、行业信息、配置项 |
| `inventory` | 商品、库存、临期、低库存预警 |
| `ledger` | 交易记录、确认入账、作废/修正 |
| `receivable-payable` | 应收应付、收款、付款、催款记录 |
| `ai-orchestration` | 调度 AI 子服务，接收解析结果 |
| `file` | 上传凭证、对象存储元数据管理 |
| `sync` | 客户端同步、幂等处理、冲突处理 |
| `backup` | 备份记录、恢复任务 |
| `notification` | 短信、站内消息、推送通知 |
| `audit` | 操作日志、安全日志、审计追踪 |

### 5.3 后端分层建议

```text
controller
  -> application service
    -> domain service
      -> repository
        -> PostgreSQL / Redis / MQ / Object Storage
```

### 5.4 后端关键原则

- 涉及账目确认、库存变更、应收应付余额更新的操作必须在同一事务内完成
- 所有“确认写入”类接口必须支持幂等
- 删除操作默认采用“逻辑作废”而不是物理删除
- AI 解析结果不得直接入账，必须经过确认接口
- 客户端同步必须带 `deviceId`、`requestId`、`idempotencyKey`

## 六、AI 子服务架构

### 6.1 AI 子服务职责

AI 子服务建议独立部署，职责包括：

- 语音转文字（STT）
- 图片 OCR 识别
- 结构化字段提取
- LLM 意图识别与多轮补问建议
- 风险评分与置信度分级

### 6.2 AI 子服务不负责

- 不直接写入账本数据库
- 不直接修改库存
- 不直接更新待收待付余额

### 6.3 AI 主流程

```text
客户端上传文本/语音/图片
  -> 后端创建解析任务
    -> 投递 RabbitMQ
      -> AI Service 消费任务
        -> 调用 STT / OCR / LLM
          -> 返回结构化结果 + 置信度
            -> 主后端保存解析结果
              -> 客户端展示确认卡片
                -> 用户确认后正式入账
```

## 八、接口设计原则

### 8.1 接口风格

- 协议：`HTTPS`
- 风格：`RESTful + JSON`
- 版本前缀：`/api/v1`
- 鉴权方式：`Bearer Token`

### 8.2 返回结构建议

```json
{
  "code": 0,
  "message": "ok",
  "requestId": "req_20260414_xxx",
  "data": {}
}
```

### 8.3 通用约定

- 时间字段统一返回 ISO 8601 格式
- 金额字段统一使用字符串或定点数，不使用前端浮点直接做账务计算
- 所有写接口支持 `Idempotency-Key`
- 所有分页接口统一使用：
  - `page`
  - `pageSize`
  - `total`
  - `list`

### 8.4 错误码建议

| 错误码 | 含义 |
|-------|------|
| `0` | 成功 |
| `4001` | 参数错误 |
| `4002` | 验证码错误 |
| `4003` | 登录状态失效 |
| `4004` | 无权限访问 |
| `4005` | 幂等冲突/重复提交 |
| `4006` | 资源不存在 |
| `4007` | 库存不足 |
| `4008` | 数据冲突 |
| `5001` | 系统内部错误 |
| `5002` | 第三方服务异常 |

## 十、部署架构建议

### 10.1 环境规划

- `dev`：开发环境
- `test`：测试环境
- `staging`：预发布环境
- `prod`：生产环境

### 10.2 首发部署建议

```text
Nginx / API Gateway
  -> Spring Boot API
  -> FastAPI AI Worker
  -> PostgreSQL
  -> Redis
  -> RabbitMQ
  -> MinIO
  -> Prometheus + Grafana
```

### 10.3 部署原则

- 首发先用 `Docker Compose` 或等效容器编排
- 业务稳定后再考虑 `Kubernetes`
- 数据库和对象存储优先保证备份与恢复能力
- 所有核心服务必须具备健康检查

## 十一、POC 验证清单

在正式锁定移动端技术栈前，建议先完成以下 POC：

1. `uni-app x` 编译到 iOS / Android / 微信小程序三端
2. 登录与验证码流程跑通
3. 录音、上传、解析、确认入账跑通
4. 图片拍摄、上传、OCR 解析跑通
5. 库存长列表滚动性能验证
6. iOS 输入框、数字键盘、滚动体验验证
7. 小程序样式和交互差异验证
8. 生物识别或密码兜底链路验证

**POC 通过标准**：
- 核心链路稳定可用
- 三端主要页面交互一致性可接受
- iOS 和 Android 不卡顿、不频繁崩溃
- 小程序端无明显样式错乱和输入行为异常
